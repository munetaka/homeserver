# 2026-07-20: 日別パネルの値がタイムレンジで変わる(VM のサンプル持ち越しと評価グリッド位相)

## 概要

電力ダッシュボードの日別パネル(kWh・自家消費率)で、**同じ日の値が表示レンジによって
変わる**表示不具合。07-13 は固定14日表示だと 07-12 の完全なコピー(買電 10.9 kWh)、
ズーム表示だと 9 時間ズレた窓の集計(15.8 kWh)になり、真値(19.4 kWh)は
どちらにも表示されていなかった。ユーザーが「07-12 と 07-13 が全く同じ値」という
違和感から発見。同じクエリを使う `el cost-update` の日次料金(07-13 以降)にも波及していた。

## 根本原因(2つの独立した問題の重なり)

日別値のクエリは `sum(energy_day_kwh{kind=...}) or sum(increase(power_*_total_kwh[1d] offset -1d))`
のハイブリッド式で、実サンプルはインポート済みの 2026-07-12 まで、以降はクエリ時に
積算メーターから都度計算していた。

1. **VM のサンプル持ち越し(左辺)**: 時間窓を指定しないセレクタは、VM がサンプル間隔から
   推定した猶予時間だけ最終サンプルを持ち越す。日次系列は間隔1日 → 猶予も約1日と推定され、
   系列終端(07-12 00:00)の**ちょうど24時間後の評価点 07-13 00:00 に 07-12 の値が再登場**。
   `or` は左辺優先なので正しいフォールバック計算が隠され、07-13 が 07-12 のコピーになる。
   - 間隔推定は**クエリ範囲内のサンプル数に依存**するため、9日レンジでは再現せず
     14日以上のレンジでのみ再現する(調査を混乱させた点)
2. **評価グリッドの位相(右辺)**: Grafana は1日ステップのグリッドを日境界に丸めるが、
   閲覧方法によって JST 0時 に揃う場合と UTC 0時(JST 9時)に揃う場合がある。
   フォールバックの `increase[1d]` は評価時刻から24時間の窓なので、グリッドが 9時 に
   揃うと**集計窓ごと9時間ズレた「日別値」**になる。

インポート済みの日(実サンプルがある日)はどちらのグリッドでも同じサンプルを拾うため無事。
問題は「実サンプルの無い日をクエリ時に都度計算する」設計そのものにあった。

## 影響

- 日別パネルの 07-13 以降の表示が閲覧レンジ依存(データベース自体は無傷)
- `cost_day_yen` の 07-13 以降が持ち越し値/ズレ窓値から計算されていた可能性

## 対応

1. **確定日の実体化**: `el cost-update`(毎晩実行)が、`energy_day_kwh` の無い確定日
   (昨日まで)を積算メーターの JST カレンダー日ちょうどの増分で計算し、
   **実サンプルとして書き込む**ようにした(cost.py `build_day_kwh_lines`)。
   インポート済みの日はスキップして AiSEG2 公式値を保持。都度計算が残るのは
   進行中の「今日」の暫定表示のみ。
2. **明示窓化**: ダッシュボードと cost.py の `energy_day_kwh` セレクタを
   `last_over_time(...[12h])` に変更。12h は JST 0時打点のサンプルを両グリッド
   (0時/9時)で拾え、前日分(24h以上前)は決して入らない値。これで持ち越しが
   構造的に起きなくなる(今日=未実体化日に昨日の残像が出ることも防ぐ)。
3. 初回実行で 07-13〜07-19 を実体化し、日次料金も再計算。

## 教訓

- **VM の窓なしセレクタは「系列が終わった後も1サンプル分生き続ける」**。まばらな系列
  (日次・期間次)を `or` フォールバックの左辺に置くときは必ず明示窓を付ける
- 日別・期間別のような「確定したら不変」の集計値は、クエリ時導出ではなく
  **夜間ジョブで実体化**するとグリッド位相・staleness の問題から一括で解放される
- レンジ長で挙動が変わる不具合は VM の間隔自動推定を疑う(再現時はレンジも固定する)
